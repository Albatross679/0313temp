# Migrating Machine Learning Projects Between VMs

## 1. GitHub's Role: What Goes On GitHub vs What Doesn't

### Files That Belong on GitHub

| Category | Examples | Why |
|----------|----------|-----|
| Source code | `.py`, `.sh`, training scripts, inference code | Core intellectual property, must be versioned |
| Configuration | `requirements.txt`, `pyproject.toml`, `poetry.lock`, `environment.yml`, `Dockerfile`, `docker-compose.yml`, `.devcontainer/` | Environment reproducibility |
| Small config files | hyperparameter configs, `.yaml`/`.json` configs | Experiment reproducibility |
| Documentation | `README.md`, report LaTeX source | Project understanding |
| DVC metadata | `.dvc` files, `dvc.yaml`, `dvc.lock` | Pointers to large files stored elsewhere |
| Evaluation scripts | `evaluate.py`, metric computation | Reproducible evaluation |
| Small sample data | Tiny test fixtures (<1 MB) | Unit testing |
| `.gitignore` | One comprehensive file | Team-wide consistency |

### Files That Do NOT Belong on GitHub

| Category | Examples | Why | Where Instead |
|----------|----------|-----|---------------|
| Model weights/checkpoints | `*.pt`, `*.pth`, `*.ckpt`, `*.safetensors`, `*.h5` | Too large (100 MB - 100+ GB) | Cloud storage / W&B Artifacts / DVC remote |
| Datasets | `*.csv`, `*.parquet`, `*.db`, `*.jsonl` (large) | Too large, often sensitive | Cloud storage / DVC remote |
| HuggingFace cache | `~/.cache/huggingface/` | Enormous, machine-specific | Re-download or transfer via rsync/rclone |
| Training outputs | `output/`, `runs/`, TensorBoard logs | Ephemeral, large | W&B / MLflow / cloud storage |
| Compiled artifacts | `__pycache__/`, `*.pyc`, `.eggs/` | Generated, machine-specific | Regenerated automatically |
| Environment dirs | `venv/`, `.conda/`, `node_modules/` | Machine-specific, huge | Rebuilt from lockfiles |
| IDE settings | `.idea/`, `.vscode/settings.json` (user-specific) | Personal preference | Local only |
| Secrets | `.env`, API keys, credentials | Security risk | Environment variables / secret managers |
| OS/temp files | `.DS_Store`, `Thumbs.db`, `*.swp` | Noise | Never stored |
| LaTeX build artifacts | `*.aux`, `*.log`, `*.fls`, `*.fdb_latexmk` | Regenerated by compiler | Regenerated on build |
| Jupyter checkpoints | `.ipynb_checkpoints/` | Auto-generated | Regenerated |

## 2. Cloud Storage (Backblaze B2, S3, GCS)

### What Large Files People Store There

| Artifact Type | Typical Size | Storage Tier | Access Pattern |
|---------------|-------------|--------------|----------------|
| **Training datasets** | 1 GB - 1 TB+ | Hot/warm | Read frequently during training |
| **Model weights (final)** | 100 MB - 100 GB | Hot | Downloaded for inference/fine-tuning |
| **Checkpoints (intermediate)** | 100 MB - 100 GB each | Warm/cold | Accessed for resuming training or rollback |
| **HuggingFace model cache** | 1-50 GB per model | Hot | Transferred when setting up new VM |
| **Preprocessed features** | 1-100 GB | Warm | Reused across experiments |
| **Evaluation outputs** | 10 MB - 1 GB | Cold | Archival, occasional review |
| **W&B/MLflow artifacts** | Variable | Managed | Accessed through platform APIs |

### Backblaze B2 Specifics

- **S3-compatible API**: Works with any S3-compatible tool (DVC, rclone, boto3, AWS CLI)
- **Pricing**: ~$6/TB/month storage, free egress to partner compute providers (Vultr, etc.), 3x free egress for everyone
- **Up to 80% cheaper** than AWS S3
- **DVC integration**: `dvc remote add -d storage s3://mybucket/dvcstore` then `dvc remote modify storage endpointurl https://s3.us-west-002.backblazeb2.com`
- **B2 Overdrive**: High-throughput tier for large-scale training data pipelines (terabit/s throughput)
- **10 GB free** for testing

### Provider Comparison

| Feature | Backblaze B2 | AWS S3 | GCS |
|---------|-------------|--------|-----|
| Storage cost | ~$6/TB/mo | ~$23/TB/mo | ~$20/TB/mo |
| Egress cost | 3x free, then $0.01/GB | $0.09/GB | $0.12/GB |
| S3 compatibility | Yes (API-compatible) | Native | Interop mode |
| DVC support | Via S3 API | Native | Native |
| Free tier | 10 GB | 5 GB (12 months) | 5 GB |
| ML integrations | Vultr, CoreWeave GPU partners | SageMaker, EC2 | Vertex AI, GCE |

### DVC (Data Version Control) as the Glue

DVC is the most popular tool for bridging GitHub and cloud storage:

1. **In Git**: You commit `.dvc` files (tiny metadata/pointer files) and `dvc.lock`
2. **In cloud storage**: DVC stores the actual large files (datasets, models, checkpoints)
3. **Workflow**:
   ```bash
   # Setup (once)
   pip install 'dvc[s3]'
   dvc init
   dvc remote add -d myremote s3://my-bucket/dvc-store
   dvc remote modify myremote endpointurl https://s3.us-west-002.backblazeb2.com  # for B2

   # Track large files
   dvc add data/training_data.parquet
   dvc add model/best_checkpoint.pt
   git add data/training_data.parquet.dvc model/best_checkpoint.pt.dvc .gitignore
   git commit -m "Track training data and model checkpoint"
   dvc push  # uploads to cloud storage

   # On new machine
   git clone <repo>
   dvc pull  # downloads large files from cloud storage
   ```

### Why DVC Over Git LFS

- **Git LFS** stores files on the Git hosting provider's servers (GitHub LFS has bandwidth limits and costs)
- **Git LFS** struggles with files >10 GB and causes bandwidth throttling
- **DVC** can use any storage backend (S3, B2, GCS, Azure, SSH, local)
- **DVC** provides ML pipeline tracking (`dvc.yaml`) beyond just file storage
- **DVC** is free and open source with no per-file size limits
- **Recommendation**: Use DVC for datasets and model weights; optionally use Git LFS for moderate files (5-50 MB) like documentation images

## 3. Docker: Dockerfiles, Images, and Workflow

### What People Actually Do

The consensus is: **commit the Dockerfile to Git, push built images to a registry**. Both are used, serving different purposes.

#### Dockerfile (committed to Git)
- Acts as a **blueprint** / source of truth for the environment
- Version-controlled alongside code
- Enables **auditing**: anyone can review exactly what's in the environment
- Supports **reproducibility**: rebuild from scratch if needed

#### Docker Image (pushed to registry)
- Acts as the **pre-built, ready-to-use** environment
- Faster to deploy than rebuilding (pull vs build)
- Tagged with versions for traceability
- Pushed to Docker Hub, GitHub Container Registry (ghcr.io), NVIDIA NGC, AWS ECR, or Google Artifact Registry

### Typical ML Dockerfile Structure

```dockerfile
# Start from NVIDIA CUDA base for GPU support
FROM nvidia/cuda:12.1.0-cudnn8-runtime-ubuntu22.04

# System dependencies
RUN apt-get update && apt-get install -y \
    python3 python3-pip python3-dev git wget \
    && rm -rf /var/lib/apt/lists/*

# Python dependencies (cached layer - copy requirements first)
COPY requirements.txt /tmp/requirements.txt
RUN pip3 install --no-cache-dir -r /tmp/requirements.txt

# Application code (changes more frequently, separate layer)
COPY . /app
WORKDIR /app

# Default command
CMD ["python3", "train_t5.py", "--finetune"]
```

**Key best practices:**
1. **Copy requirements.txt before code** - Docker caches the dependency layer, so rebuilds are fast when only code changes
2. **Use slim/minimal base images** - `python:3.11-slim` for CPU-only, `nvidia/cuda:*-runtime-*` for GPU
3. **Create `.dockerignore`** - Exclude `data/`, `model/`, `output/`, `venv/`, `.git/`, `wandb/`
4. **Multi-stage builds** - Separate build dependencies from runtime
5. **Pin versions** - Both base image tags and pip package versions

### Running with GPU Support

```bash
# Requires NVIDIA Container Toolkit on host
docker run --gpus all -it --rm \
    --ipc=host \
    --ulimit memlock=-1 \
    --ulimit stack=67108864 \
    -v /data:/data \
    -v /models:/models \
    myimage:latest
```

### Devcontainers for Development

VS Code and JetBrains support `.devcontainer/devcontainer.json` which:
- Automates the entire development environment setup
- Can reference a Dockerfile or a pre-built image
- Configures GPU access, extensions, environment variables
- Enables "clone repo + open in container = ready to develop"

### Registry Options

| Registry | Best For | Cost |
|----------|----------|------|
| Docker Hub | Public images, OSS | Free for public, paid for private |
| GitHub Container Registry (ghcr.io) | GitHub-integrated projects | Free with GitHub account |
| NVIDIA NGC | Pre-built ML frameworks (PyTorch, TF) | Free |
| AWS ECR | AWS deployments | Pay per storage + transfer |
| Google Artifact Registry | GCP deployments | Pay per storage + transfer |

## 4. End-to-End Migration Workflow

### Step-by-Step: Moving an ML Project from VM-A to VM-B

#### Phase 1: Prepare on Source VM (VM-A)

```bash
# 1. Ensure all code is committed and pushed
git status
git add -A && git commit -m "Pre-migration snapshot"
git push origin main

# 2. Push large files to cloud storage via DVC
dvc add data/ model/best_checkpoint.pt
git add *.dvc .gitignore
git commit -m "Track large files with DVC"
git push
dvc push  # uploads to B2/S3/GCS

# 3. Upload model weights to W&B (if using W&B artifacts)
# (typically done automatically during training via log_model_artifact)

# 4. Push Docker image to registry (if using Docker)
docker build -t myregistry/myproject:v1.0 .
docker push myregistry/myproject:v1.0

# 5. Export any local-only artifacts
#    - HuggingFace cache (if not wanting to re-download)
rclone sync ~/.cache/huggingface/ b2:mybucket/hf-cache/ --progress
#    - Or use rsync for direct VM-to-VM transfer
rsync -avz --progress ~/.cache/huggingface/ user@vm-b:~/.cache/huggingface/
```

#### Phase 2: Set Up Destination VM (VM-B)

```bash
# 1. Install system prerequisites
sudo apt-get update && sudo apt-get install -y git python3 python3-pip python3-venv
# Install NVIDIA drivers + CUDA (if GPU VM)
# Install Docker + NVIDIA Container Toolkit (if using Docker)

# 2. Clone the repository
git clone git@github.com:user/ml-project.git
cd ml-project

# 3. Recreate Python environment
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
# OR: conda env create -f environment.yml
# OR: poetry install

# 4. Pull large files from cloud storage
pip install 'dvc[s3]'
dvc pull  # downloads datasets and model weights from B2/S3/GCS

# 5. Restore HuggingFace cache (if transferred)
# Either re-download (slow but clean):
python -c "from transformers import AutoModel; AutoModel.from_pretrained('google-t5/t5-small')"
# Or use the rclone-synced cache from Phase 1

# 6. Configure environment variables
export HF_HOME=/path/to/huggingface/cache
export WANDB_API_KEY=xxx
export CUDA_VISIBLE_DEVICES=0

# 7. Verify everything works
python train_t5.py --finetune --max_n_epochs 1  # quick smoke test
```

#### Phase 2 (Docker Alternative)

```bash
# 1. Install Docker + NVIDIA Container Toolkit on VM-B
# 2. Pull the pre-built image
docker pull myregistry/myproject:v1.0
# 3. Pull data separately (data is NOT in the image)
git clone git@github.com:user/ml-project.git
cd ml-project && dvc pull
# 4. Run with mounted volumes
docker run --gpus all -it --rm \
    -v $(pwd)/data:/app/data \
    -v $(pwd)/model:/app/model \
    -v ~/.cache/huggingface:/root/.cache/huggingface \
    myregistry/myproject:v1.0
```

### What Goes Where - Summary Diagram

```
GitHub Repository (code + config + pointers)
    ├── Source code (.py, .sh)
    ├── Config files (requirements.txt, Dockerfile, .devcontainer/)
    ├── DVC metadata (.dvc files, dvc.yaml, dvc.lock)
    ├── Documentation (README, report/)
    └── .gitignore

Cloud Storage - Backblaze B2/S3/GCS (large files)
    ├── Training datasets
    ├── Model weights / checkpoints
    ├── Preprocessed features
    └── HuggingFace model cache (optional backup)

Docker Registry (pre-built environments)
    └── Docker images with OS + CUDA + Python + dependencies
        (code and data are NOT baked into images — mounted at runtime)

W&B / MLflow (experiment tracking + artifacts)
    ├── Training metrics and logs
    ├── Model artifacts (versioned weights)
    ├── Hyperparameter configs per run
    └── Evaluation results
```

## 5. Best Practices

### Model Weights

- **Never commit to Git** — too large, bloats repo permanently
- **Use W&B Artifacts** for versioned model storage linked to training runs
- **Use DVC** to track model files with pointer files in Git
- **Use `.safetensors` format** over `.pkl`/`.pt` when possible (safer, no arbitrary code execution)
- **Document the base model and dependencies** (framework version, model architecture) alongside weights
- **Checkpoints should include**: model state_dict, optimizer state, epoch number, best metric, config used

### Datasets

- **Small datasets (<50 MB)**: Can go in Git (or Git LFS)
- **Medium datasets (50 MB - 10 GB)**: DVC with cloud storage backend
- **Large datasets (>10 GB)**: Cloud storage (B2/S3/GCS) with DVC pointers or direct access
- **Immutable raw data**: Never modify original data; keep transformations in code
- **Three-folder convention**: `data/raw/` (immutable), `data/interim/` (intermediate), `data/processed/` (training-ready)

### Checkpoints

- **Save both "best" and "last"** checkpoints
- **Include metadata**: epoch, step, validation metric, config hash
- **Upload to cloud storage** periodically (not just at end of training)
- **For large models**: Use sharded checkpoints, async saving, and multi-tier checkpointing (local SSD first, then cloud)

### Dependencies & Environment Reproducibility

| Approach | Pros | Cons | Best For |
|----------|------|------|----------|
| `requirements.txt` (pinned) | Simple, universal | No lock file, no transitive deps by default | Quick projects |
| `poetry` + `poetry.lock` | Deterministic, hash-verified | Python-only, no CUDA/system deps | Python-focused projects |
| `conda` + `environment.yml` | Handles CUDA, system libs, non-Python deps | Slower solving, less reproducible without `--export` | GPU/scientific projects |
| `Docker` | Complete environment (OS + CUDA + Python + deps) | Heavier, requires Docker knowledge | Production, team collaboration |
| `Docker` + `poetry`/`conda` inside | Best of both worlds | Most setup effort | Maximum reproducibility |
| `devcontainer` | One-click dev environment | IDE-specific (VS Code/JetBrains) | Development workflows |

**Recommended stack**: Dockerfile (committed to Git) + `requirements.txt` or `poetry.lock` inside + pre-built image pushed to registry

### Transfer Tools

| Tool | Best For | Notes |
|------|----------|-------|
| `rclone` | Cloud-to-cloud transfers, B2/S3 sync | Multi-threaded, supports 70+ backends |
| `rsync` | VM-to-VM direct transfers | Single-threaded but delta-efficient |
| `scp`/`sftp` | Simple one-off file transfers | No delta sync, no cloud support |
| DVC push/pull | Versioned dataset/model transfers | Integrated with Git workflow |
| W&B artifacts | Model checkpoint transfers | Linked to experiment tracking |
| `huggingface-cli` | HF model cache management | `huggingface-cli download` on target VM |

### Security

- **Never commit secrets** (`.env`, API keys, credentials) to Git
- Use environment variables or secret managers (AWS Secrets Manager, HashiCorp Vault)
- Use `.dockerignore` to exclude secrets from Docker images
- Use SSH keys (not passwords) for VM access and Git authentication
- Audit Docker images for vulnerabilities before deploying
