## Summary

The DPO sweep (`part1/dpo_sweep.py`) crashed on every trial with a CUDA device-side assert in `ScatterGatherKernel.cu`. The first trial failed at batch 179/1226, and subsequent trials failed immediately because the CUDA context was poisoned.

## Error

```
/pytorch/aten/src/ATen/native/cuda/ScatterGatherKernel.cu:163: operator():
  Assertion `idx_dim >= 0 && idx_dim < index_size && "scatter gather kernel index out of bounds"` failed.

Trial failed: CUDA error: device-side assert triggered
```

W&B killed the sweep after 5 consecutive failures (`WANDB_AGENT_MAX_INITIAL_FAILURES`).

## Root Cause

`compute_restricted_log_probs()` in `part1/dpo_loss.py` calls `model.remap_targets(decoder_targets)` which maps original T5 token IDs to restricted vocab indices. Tokens not in `FlightSQLVocab` get mapped to `-1`. This `-1` index is then passed to `torch.gather()`, causing the out-of-bounds CUDA assert.

The DPO preference data contains **rejected** SQL strings generated by the model during candidate generation (`part1/dpo_data.py`). Some of these rejected candidates contain garbled token fragments not present in the training SQL data, and therefore not in the restricted vocabulary.

## Impact

- 15 out of 9,801 triplets (0.2%) had OOV tokens, all in the rejected SQL
- 20 total OOV token occurrences across 19 unique token IDs
- Example OOV tokens: `"Frank"`, `"Flight"`, `"IP"`, `"BL"`, `"RES"`, `"ARA"` — subword fragments from bad model outputs

## Fix

In `part1/dpo_loss.py`, `compute_restricted_log_probs()`:

1. Detect unmapped tokens: `unmapped_mask = (remapped_targets == -1)`
2. Clamp to valid index: `remapped_targets = remapped_targets.clamp(min=0)`
3. Zero out unmapped positions in the loss mask: `mask = mask * (~unmapped_mask).float()`

This ensures `torch.gather` always receives valid indices, and unmapped tokens contribute zero to the sequence log probability — equivalent to treating them as if they don't exist in the sequence.

## Verification

Relaunched sweep (ID `igz8embc`) with fix. First trial passed batch 179 (previous crash point) and continued training normally.
