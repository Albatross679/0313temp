# SQL Exact Match Tokenizer Artifact

## Summary

SQL exact match (SQL EM) for T5 models is reported at ~2-3%, despite Record F1 of ~62% and Record EM of ~59%. Investigation reveals this is predominantly a **measurement artifact** caused by T5's SentencePiece tokenizer decode behavior, not a genuine model deficiency.

## Root Cause

### The Problem

The ground truth SQL (`data/dev.sql`, `data/train.sql`) consistently uses ` , ` (space-comma-space) for all comma separators:

```sql
SELECT DISTINCT flight.flight_id FROM flight flight_1 , airport airport_1 , city city_1
WHERE city_1.city_name = 'DENVER' AND airport_1.city_code = city_1.city_code
```

The T5 SentencePiece tokenizer tokenizes ` , ` as two tokens: `'_'` (space sentinel) + `','` (comma). During `batch_decode(outputs, skip_special_tokens=True)`, the leading space sentinel `'_'` before `,` gets collapsed by the SentencePiece decode algorithm, producing `', '` (comma-space, no preceding space):

```sql
SELECT DISTINCT flight.flight_id FROM flight flight_1, airport airport_1, city city_1
WHERE city_1.city_name = 'DENVER' AND airport_1.city_code = city_1.city_code
```

### Why This Breaks SQL EM

`compute_sql_exact_match()` in `utils.py` (lines 143-153) performs pure string equality:

```python
def compute_sql_exact_match(ground_truth_queries, model_queries):
    ...
    for gt_q, model_q in zip(ground_truth_queries, model_queries):
        if gt_q == model_q:
            correct += 1
    ...
```

Since virtually all SQL queries in the dataset contain at least one comma (FROM clauses with JOINs, SELECT with multiple columns, WHERE with multiple conditions), the decode artifact causes nearly every query to fail string equality, regardless of semantic correctness.

### Evidence: Tokenizer Roundtrip Test

```python
from transformers import T5Tokenizer
tokenizer = T5Tokenizer.from_pretrained("google-t5/t5-small")

# Ground truth uses " , "
original = "FROM flight flight_1 , airport airport_1"
tokens = tokenizer.encode(original, add_special_tokens=False)
decoded = tokenizer.decode(tokens, skip_special_tokens=True)

print(f"Original: '{original}'")
# Original: 'FROM flight flight_1 , airport airport_1'
print(f"Decoded:  '{decoded}'")
# Decoded:  'FROM flight flight_1, airport airport_1'
print(f"Match: {original == decoded}")
# Match: False
```

The `'_'` token before `','` is consumed during decode, collapsing ` , ` to `, `.

### The Model IS Learning Correctly

This is critical: the model learns to produce the correct ` , ` pattern. The token sequence generated by the model includes the `'_'` before `','`. The information loss happens exclusively during the decode step (`batch_decode`), which is post-inference.

Training data confirmation:
- `data/train.sql`: 22,077 instances of ` , ` (space-comma-space), 0 instances of `, ` without leading space
- `data/dev.sql`: 2,352 instances of ` , `, 0 instances of `, ` without leading space

## Quantified Impact

| Metric | Raw (with artifact) | After Comma Normalization | Artifact Contribution |
|--------|---------------------|--------------------------|----------------------|
| T5 fine-tune SQL EM | 0.034 (16/466) | 0.725 (338/466) | 322/450 mismatches are purely comma spacing |
| T5 from-scratch SQL EM | 0.032 (15/466) | 0.504 (235/466) | Similar pattern |
| LLM prompting SQL EM | 0.000 (0/466) | 0.000 (0/466) | Different issue (wrong query structure) |

For the fine-tuned T5 model: **322 out of 450 mismatches** (71.6%) are caused solely by the comma spacing difference. The remaining 128 represent genuine semantic differences (wrong query structure, wrong predicates, etc.).

## Fix Options

### Option A: Post-process decode output (Recommended)

Apply regex to restore space before commas after `batch_decode`:

```python
import re

decoded = tokenizer.batch_decode(outputs, skip_special_tokens=True)
# Restore space before commas (SentencePiece collapses it)
fixed = [re.sub(r'(?<=[a-zA-Z0-9_)])\s*,\s*', ' , ', s) for s in decoded]
```

This is safe because SQL in this dataset never uses `, ` without a preceding space.

**Where to apply:** `_generate_predictions()` in `part1/train.py` line 217-218, after `batch_decode` call.

### Option B: Raw token-to-string conversion

Instead of `batch_decode`, manually convert token IDs to strings preserving the `'_'` sentinel as a space. More complex and fragile.

## Recommendation

Apply Option A (regex post-processing) in the prediction pipeline. This:
1. Fixes the measurement artifact without changing the model
2. Is trivially reversible
3. Produces SQL EM numbers that accurately reflect model quality
4. Is safe for this dataset (no false positives from the regex pattern)

## Affected Code
