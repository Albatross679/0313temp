  <name>Task 1: Update error analysis script with 5-category classification</name>
  <files>script/error_analysis.py</files>
  <action>
Rewrite `script/error_analysis.py` to produce the 5 error categories identified in the research, with clean report-ready output.

The script must:
1. Load ground truth SQL from `data/dev.sql`, NL from `data/dev.nl`, and ground truth records from `records/ground_truth_dev.pkl`
2. Load predictions for all 3 models: `results/t5_ft_dev.sql` + `records/t5_ft_dev.pkl`, `results/t5_scr_dev.sql` + `records/t5_scr_dev.pkl`, `results/llm_dev.sql` + `records/llm_dev.pkl`
3. For each of the 466 queries per model, classify into exactly ONE of these 5 categories (priority order for mutual exclusivity):

   **Category 1: Missing comparison operator** — T5 models drop `<`/`>` operators due to SentencePiece tokenizer artifact. Detect with regex: `\w+\.\w+\s{2,}\d+` in predicted SQL (two+ spaces between column name and number where operator should be). Also check error message for "near" + "syntax error".

   **Category 2: Query truncation / incomplete generation** — LLM dominant. Detect: `len(pred_sql) < 50` OR predicted SQL has no FROM clause OR error message contains "incomplete". For the LLM, the vast majority of "no such column" errors are actually caused by truncation (no FROM clause defines the alias).

   **Category 3: Wrong table/column reference** — Predicted SQL references undefined aliases or non-existent columns. Detect from error message: "no such column" or "no such table" (AFTER excluding truncation cases). Only count for non-truncated queries.

   **Category 4: Wrong JOIN structure** — Query executes successfully but returns wrong records due to incorrect table joins, missing table joins, or wrong subquery structure. Detect: query executes without error AND records do not match ground truth AND structural analysis shows different FROM tables or missing/extra JOINs compared to ground truth.

   **Category 5: Wrong predicate values** — Query executes successfully, correct structure, but wrong literal values in WHERE (wrong city, date, time). Detect: query executes, records don't match, but FROM/JOIN structure matches ground truth. Difference is in WHERE clause values.

   Priority order for classification: truncation > missing operator > wrong reference > (for executable queries) wrong JOIN > wrong predicate > other.

4. Output a clean summary per model in this format:
   ```
   === T5 Fine-tuned ===
   Category 1 (Missing operator): COUNT/466
   Category 2 (Truncation): COUNT/466
   ...
   Correct (exact SQL match): COUNT/466
   Correct (different SQL, same records): COUNT/466
   ```

5. For each category, output the BEST concrete example showing: (a) NL input, (b) ground truth SQL fragment, (c) predicted SQL fragment, (d) what went wrong. Select examples that are short and clearly illustrate the error.

6. Print a cross-model summary table at the end.

- Keep the script standalone with only stdlib imports (re, pickle, sqlite3, collections, pathlib)
- Add project root to sys.path so it can be run from project root: `python3 script/error_analysis.py`

- Relevant Models: T5 fine-tuned, ICL

- Relevant Models: T5 fine-tuned, T5 from scratch, ICL
