  <action>
Launch both RL training runs sequentially using nohup (per project convention for long-running jobs).

**Pre-flight check:** Verify no other training process is running:
```bash
ps aux | grep -E "python.*train" | grep -v grep
```

**Verify base checkpoint exists:** Check the path in T5GRPOConfig.base_checkpoint_path. If the DPO model from Phase 2 exists and is better, use that. Otherwise use the SFT checkpoint. The config defaults to the SFT checkpoint path.

**Run 1 -- GRPO (primary, ~1 hour budget):**
```bash
PYTHONUNBUFFERED=1 nohup python part1/rl_train.py \
  --rl_algorithm grpo \
  --max_wall_clock_hours 1.0 \
  > output/grpo_train.txt 2>&1 &
```

Monitor with `tail -f output/grpo_train.txt`. Watch for:
- Training starting (epoch 0 loss printed)
- Reward statistics (mean_reward, zero_std_frac)
- Dev F1 after first eval epoch
- Gradient norm stability

If `zero_std_frac > 0.5` after first few epochs, this means reward is too sparse (most groups have all-same reward). Note this for the report but continue -- it's a valid negative result.

If gradient norms spike repeatedly (spike detection fires), note this for report analysis.

**Wait for GRPO to complete** (early stopping or time budget).

**Run 2 -- CISPO (~1 hour budget):**
```bash
PYTHONUNBUFFERED=1 nohup python part1/rl_train.py \
  --rl_algorithm cispo \
  --max_wall_clock_hours 1.0 \
  > output/cispo_train.txt 2>&1 &
```

Same monitoring approach. Watch particularly for gradient norm instability (known CISPO issue from research, MiniMax-M1 GitHub Issue #31).

**Collect results:** After both runs complete, extract from logs:
- Best dev Record F1 for GRPO
- Best dev Record F1 for CISPO
- Number of epochs trained
- Notable observations (reward sparsity, gradient spikes, convergence behavior)
- Compare to SFT/DPO baseline dev F1

Document the experiment in `experiments/rl-fine-tuning-grpo-cispo.md` with proper frontmatter (type: experiment, status: complete).
