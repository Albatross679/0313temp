  <action>
Launch all three RL training runs in a single sequential auto-batch process.

**Pre-flight check:**
```bash
ps aux | grep -E "python.*train" | grep -v grep
script/gpu-lock.sh status
```

**Verify base checkpoint exists:** Check the path in T5GRPOConfig.base_checkpoint_path.

**Launch single process (sequential auto-batch -- all three configs):**
```bash
PYTHONUNBUFFERED=1 nohup python part1/rl_train.py \
  --max_hours 3.0 \
  > output/rl_sequential_train.txt 2>&1 &
```

This runs PPO -> GRPO -> CISPO sequentially in one process. Each config gets early stopping as its primary stop mechanism. The `--max_hours 3.0` is the overall sweep budget across all three configs. VRAM is cleaned between configs automatically.

**Start monitoring immediately after launch:**
```
/loop 10m /babysit-training
```

The babysit-training skill handles: process alive check, loss/metric trending, GPU utilization, checkpoint integrity, W&B connectivity, hung process detection.

**Monitor with tail in parallel:**
```bash
tail -f output/rl_sequential_train.txt
```

**Watch for per-config:**
- PPO: value_loss decreasing, entropy staying reasonable (not collapsing to 0)
- GRPO: clip_fraction < 0.3, KL not exploding > 10
- CISPO: gradient norm stability (known issue from MiniMax-M1 GitHub #31)
- All: zero_std_frac (reward sparsity) — if > 0.5, note for report but continue
- All: unique_completions_per_query — if = 1, mode collapse detected

**If a config crashes mid-training:**
- The sequential auto-batch continues to the next config after VRAM cleanup
- The crashed config's partial results are still in W&B
- Document the crash in the experiment doc

**Collect results after all three complete:**
Extract from W&B / logs:
- Best dev Record F1 for each algorithm (PPO, GRPO, CISPO)
- Number of epochs trained per config
- Convergence behavior (early stopping vs time budget)
- Notable observations per algorithm:
  - PPO: value head accuracy, advantage from learned baseline quality
  - GRPO: reward sparsity (dead groups), clip behavior
  - CISPO: gradient norm stability, clip behavior differences from GRPO
- Compare all three to SFT/DPO baseline dev F1

**Content (adapt based on actual results):**

1. **Methodology paragraph:**
   - Explored three online RL algorithms: PPO (Schulman et al., 2017), GRPO (DeepSeekMath, 2024), and CISPO (MiniMax-M1, 2025)
   - Applied to best available T5-base checkpoint via LoRA (r=32, alpha=64, q/k/v/o projections)
   - Group sampling: G=8 completions per query with constrained SQL decoding
   - Execution-based reward: graded (+1.0 exact match, +0.5 partial, -0.5 wrong, -1.0 error)
   - PPO uses a learned value head (MLP on mean-pooled encoder states) as baseline; GRPO/CISPO use group-relative normalization
   - PPO: clipped surrogate + value loss + entropy bonus; GRPO: clipped surrogate only; CISPO: detached IS weight clipping
   - Note this is novel work: no published GRPO/CISPO/PPO work on T5 encoder-decoder for NL-to-SQL exists
