## Directory Purposes

**`src/`:**
- Purpose: Shared infrastructure used by all three parts
- Contains: Config base classes, W&B wrapper, GPU serialization lock, system metrics
- Key files: `src/config.py`, `src/wandb_utils.py`, `src/utils/gpu_lock.py`

**`part1/`:**
- Purpose: T5 fine-tune approach — all implementation for Assignment Part 1
- Contains: Config variants (11+ experiment classes), dataset, model wrappers, full training loop with DPO support
- Key files: `part1/config.py`, `part1/train.py`, `part1/model_flightdb.py`

**`part2/`:**
- Purpose: T5 from-scratch approach — same architecture as Part 1 but random init
- Contains: Config variants, dataset (mirrors part1), training loop with multi-config sequential support
- Key files: `part2/config.py`, `part2/train.py`

**`part3/`:**
- Purpose: LLM prompting approach — Gemma 2B / CodeGemma 7B with k-shot ICL
- Contains: Config variants (k-shot, BM25, ablations, CodeGemma), raw text data loading, model init
- Key files: `part3/config.py`, `part3/data.py`, `part3/model.py`

**`data/`:**
- Purpose: Read-only dataset files — protected, must not be modified
- Contains: NL/SQL paired data for train/dev/test; SQLite flight database; JSON schema
- Key files: `data/flight_database.db`, `data/flight_database.schema`, `data/{train,dev}.{nl,sql}`

**`records/`:**
- Purpose: Pickled database record lists for evaluation caching
- Contains: Ground truth dev records (canonical), generated records from all runs
- Key files: `records/ground_truth_dev.pkl` (protected — do not modify)

**`results/`:**
- Purpose: SQL prediction output files for evaluation and submission
- Contains: `.sql` prediction files (one query per line), `.txt` log files

**`output/`:**
- Purpose: Per-run training artifacts
- Contains: Timestamped run directories with checkpoints, config, metrics
- Generated: Yes. Committed: Only final checkpoints via Git LFS.

**`script/`:**
- Purpose: Standalone utility and batch-run scripts
- Contains: GPU lock shell wrapper, monitoring scripts, experiment batch runners, analysis tools

## Key File Locations

**Entry Points:**
- `train_t5.py`: Part 1 training (fine-tune, delegates to `part1.train.main()`)
- `part2/train.py`: Part 2 training (from-scratch, run directly)
- `prompting.py`: Part 3 LLM prompting
- `evaluate.py`: Standalone evaluation CLI

**Configuration:**
- `src/config.py`: Base config hierarchy (`BaseConfig`, `SLNeuralConfig`, `SLNeuralClsConfig`)
- `part1/config.py`: All T5 fine-tune experiment variants
- `part2/config.py`: All T5 from-scratch experiment variants
- `part3/config.py`: All prompting experiment variants

**Core Logic:**
- `part1/train.py`: Complete training loop (train_epoch, eval_epoch_gpu, eval_epoch_sql, test_inference)
- `part1/model_flightdb.py`: FlightSQLVocab, T5ForFlightSQL restricted-vocab wrapper
- `utils.py`: Metrics computation (Record F1, Record EM, SQL EM), SQL execution with ThreadPoolExecutor
- `prompting.py`: create_prompt(), exp_kshot() — core prompting logic
- `prompting_utils.py`: read_schema(), extract_sql_query() — schema loading and SQL extraction

**Infrastructure:**
- `src/wandb_utils.py`: All W&B interactions (setup_run, log_epoch_metrics, log_model_artifact)
- `src/utils/gpu_lock.py`: GpuLock context manager
- `t5_utils.py`: AdamW optimizer + LR scheduler factory (delegates to part1)

**Testing/Evaluation:**
- `part1/eval_checkpoint.py`: Interactive checkpoint evaluator with per-example error analysis
- `part2/eval_checkpoint.py`: Same for Part 2 checkpoints

## Naming Conventions

**Files:**
- Python modules: `snake_case.py`
- Config variants: `{BaseClass}_{variant_name}` (e.g., `T5FineTuneConfig_lora_v1`)
- Checkpoint files: `model_best.pt`, `model_last.pt`, `training_state.pt`
- SQL prediction files: `{part}_{experiment}_{split}.sql` (e.g., `t5_ft_dev.sql`, `llm_k3_dev.sql`)
- Record files: same prefix with `.pkl` extension
- Run directories: `output/{config_name}_{YYYYMMDD_HHMMSS}/`

**Directories:**
- Part directories: `part{N}/` (lowercase, numeric suffix)
- Shared infra: `src/`
- Output artifacts: `output/`, `results/`, `records/`
- Documentation: `experiments/`, `issues/`, `knowledge/`, `logs/`, `tasks/`

## Where to Add New Code

**New Training Config Variant (T5 fine-tune):**
- Add `@dataclass class T5FineTuneConfig_{name}(T5FineTuneConfig)` to `part1/config.py`
- Override only the fields that differ from the parent class
- Invoke with `python train_t5.py --config T5FineTuneConfig_{name}`

**New Training Config Variant (T5 from-scratch):**
- Add `@dataclass class T5ScratchConfig_{name}(T5ScratchConfig)` to `part2/config.py`
- Invoke with `python part2/train.py --config T5ScratchConfig_{name}`

**New Prompting Variant:**
- Add `@dataclass class {Name}Config(PromptingConfig)` to `part3/config.py`
- Invoke with `python prompting.py --config {Name}Config` (via `part3/train.py`)

**New Shared Infrastructure:**
- Utilities: `src/utils/{module_name}.py`
- Config additions: extend `src/config.py` hierarchy

**New Feature in Model Pipeline:**
- Data changes: `part1/data.py` (and mirror in `part2/data.py` if applicable)
- Model architecture: `part1/model.py` or `part1/model_flightdb.py`
- Training loop changes: `part1/train.py`

**New Experiment Scripts:**
- Batch runners: `script/run_{experiment}.sh`
- Analysis tools: `script/{analysis_name}.py`

**New Utility Scripts (standalone analysis):**
- `script/` directory for one-off analysis tools
